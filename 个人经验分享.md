# AI 编程实践：个人经验分享

本文只分享研发中的实际用法，目标是提高效率、减少返工，并让 AI 更稳定地产出可用结果。

## 一、先说结论

1. **先拆问题，再让 AI 写代码。**
2. **模块边界比具体实现更重要。**
3. **复杂任务应由多个 Agent 分工完成。**
4. **测试和 Review 必须与实现解耦。**
5. **AI 结果必须经过真实环境验证。**

AI 擅长解决边界清晰的局部问题。任务越大、上下文越长、跨模块联动越多，结果越不稳定。

所以从接到需求开始，就要思考如何模块化分工，而不是等到开发过程中再拆。模块越小、边界越清楚，AI 在小上下文内完成任务的成功率越高。

---

## 二、推荐工作流

```text
需求澄清
  ↓
AI 多角度提问
  ↓
梳理业务流程、数据流、异常场景和边界条件
  ↓
输出并评审设计文档
  ↓
拆分模块、定义接口
  ↓
AI 分模块实现
  ↓
独立测试与 Review
  ↓
真实环境验证
```

复杂需求的第一轮，不要让 AI 直接写代码。可以先使用以下提示词：

```text
请先不要写代码，也不要直接给方案。

请从架构、测试、运维、安全和产品角度分析需求，并持续向我提问。

重点确认：
1. 业务流程
2. 数据流
3. 模块划分
4. 边界条件
5. 异常处理
6. 潜在风险

信息充分后，再输出 design.md。
```

AI 在需求阶段的重要价值，不只是回答问题，更是发现遗漏的问题。

## 三、工程设计原则

### 1. 从需求阶段开始拆模块

不要等到编码阶段再拆分。每个模块应满足：

- 输入明确
- 输出明确
- 职责单一
- 可独立实现和验证

边界清晰后，AI 成功率更高，也更适合多人或多 Agent 并行开发。

这里的重点不是简单把代码拆开，而是先把业务问题拆成多个 AI 能独立理解、独立实现、独立验证的小任务。

### 2. 适度重复，谨慎抽象

在一定范围内，重复造小轮子可能比全局抽象、高度复用更划算。

AI 更擅长复制、修改和局部优化，不擅长复杂继承、深层抽象和跨模块联动。为了追求统一而过早抽象，反而会增加上下文长度和理解成本，降低 AI 产出质量。

例如两个差异较大的协议解析器，可以先独立实现：

```text
GBT32960Parser
JT808Parser
```

不必过早抽象为统一基类。建议：

- 小型、局部逻辑：允许适度重复
- 较大且通用的能力：封装为独立模块，并提供标准化 API
- 出现明确复用需求后：再进行抽象

抽象的目标是降低复杂度，而不是消除所有重复。

## 四、多 Agent 协作方式

复杂项目不要依赖一个长对话完成全部工作。上下文过长容易导致目标偏移，也不利于独立验证。

可以按角色拆分：

| 角色 | 主要职责 |
| --- | --- |
| 设计 Agent | 调研资料、理解规范、输出总体设计 |
| 接口 Agent | 定义模块边界和接口，不负责具体实现 |
| 实现 Agent | 按设计分模块编码，不随意修改架构 |
| 测试 Agent | 根据需求、规范和设计独立编写测试 |
| Review Agent | 检查功能、性能、内存、安全和可维护性 |
| 优化 Agent | 根据 Review 结果修改并回归验证 |

核心原则：**把 AI 当作分工明确的团队，而不是全能工程师。**

### 测试和 Review 必须独立agent

不要让同一个 Agent 同时完成设计、实现、测试和 Review。否则它很容易沿用同一套理解，自证正确。

```text
代码通过测试或 Review ≠ 代码满足真实需求
```

测试与 Review 应优先依据需求、规范和设计，再检查实现细节。

---

## 五、AI 结果仍需真实环境验证

AI 可以生成代码和测试，但无法穷举真实环境中的问题。上线前仍需重点验证：

- 内存泄漏
- 竞争条件和锁问题
- NUMA 与缓存问题
- 性能退化
- 异常数据和异常流量
- 压力测试与长时间稳定性

AI 负责提高产出速度，工程师负责确认结果可信。

---

## 六、AI 不只是代码生成器

AI 还适合用于：

- 需求分析：补充场景、发现遗漏、识别风险
- 方案设计：架构讨论、模块划分、接口设计
- 文档整理：设计文档、测试文档、会议纪要、汇报材料
- 图形生成：架构图、时序图、PPT 素材
- 工程检查：代码 Review、性能分析、安全检查

研发人员的核心能力也在变化。相比单纯提高编码速度，更重要的是：

- 拆解问题
- 定义边界
- 表达需求
- 判断取舍
- 验证结果

**最终目标不是让 AI 替我们思考，而是把复杂问题拆成 AI 能稳定完成的任务，并把更多精力放在设计、判断和验证上。**
