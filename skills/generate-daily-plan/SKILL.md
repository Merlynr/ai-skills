# 今日计划生成器

## 目标

生成或更新 `10 Daily/YYYY-MM-DD.md`，把笔记里的待办、历史未完成任务和用户新增要求整理成当天可执行计划。

## 工作流

1. 先确定目标日期，默认使用今天的本地日期（Asia/Shanghai）。
2. 先分出 `笔记任务池`：
   - `20 Projects/**` 中的未完成任务、阶段计划、项目看板任务。
   - `10 Daily/**` 中今天以前遗留的未完成任务。
   - 其中，优先回收最近一篇 `今日计划` 和 `Carry-over` 里仍未完成的任务，再继续向更早的日计划回收。
   - `00 Inbox/**` 中尚未清空的收件箱事项。
   - 默认跳过 `20 Projects/Stock Prompt+report` 和 `20 Projects/龟龟策略`，它们不是笔记类内容。
   - 把候选项整理成一个独立任务池，先不要混入用户输入。
3. 再分出 `用户任务池`：
   - 用户本次输入的新任务、目标、时间限制、偏好、顺序要求。
   - 如果用户明确指定顺序，就优先保留该顺序。
4. 分别整理两个任务池：
   - 过期任务
   - 今日截止或今日安排任务
   - 需要延续的旧任务
   - 其余可选任务
   - 同一项目或同一阶段计划里，先选最早漏掉的未完成项，再选今天到期的项。
   - 今天以前的 `今日计划` 里仍未完成的任务要优先回收，且要完整回收到当天计划里，不再受 3 条限制。
   - `Carry-over` 的 3 条上限只约束项目计划中的旧欠账，不约束 daily 遗留任务。
   - 项目计划中的旧欠账在 `Carry-over` 中最多只展示 3 条，优先顺序为：最近一篇 daily 的遗留项 > 更早 daily 的遗留项 > 项目计划中的更老逾期项。
5. 最后合并成 `今日计划`：
   - 先放 `笔记任务池` 里今天以前的 daily 遗留任务到 `Carry-over`，不设 3 条上限。
   - 再把 `笔记任务池` 里最该先补的 3 条项目旧任务补进 `Carry-over`。
   - 再放 `笔记任务池` 里今天最重要的任务。
   - 再放 `用户任务池` 中今天必须处理的任务。
   - 如果空间不够，先保留笔记中的逾期任务，再保留用户任务。
   - `今日计划` 的展示优先级高于 `Carry-over`，页面上先看今天任务，再看旧欠账。
   - `Carry-over` 名额有限时，优先保留 daily 遗留任务；如果 daily 遗留已完整回收，再用剩余名额放项目逾期任务。
6. 在最终计划里必须显式列出来自笔记的未完成任务；如果没有可列出的笔记任务，也要写 `- 无`，不能留空。
7. 只写入今天真正能执行的内容，不要把所有候选项原样堆进去。
8. 新建文件时，优先参考 `90 Templates/Daily.md` 的结构，但把 Templater 占位符替换成实际日期和内容。
   - 如果生成的稳定格式比模板更好，就把改进回写到模板，保持二者一致。
9. 如果文件已存在，只更新受管理的计划区块，不要覆盖用户手写的复盘、笔记和补充内容。

## 输出规范

- 文件路径：`10 Daily/YYYY-MM-DD.md`
- 保留清晰的 frontmatter
- 用 Markdown 任务列表表示可执行事项
- 建议区块：
  - `## Focus`
  - `## 今日计划`
  - `## Carry-over`
  - `## Notes`
  - `## Review`
- 用 `<!-- DAILY-PLAN:BEGIN -->` 和 `<!-- DAILY-PLAN:END -->` 包住可重写的计划区块。
- `Carry-over` 里放从笔记中续接下来的未完成任务，其中 daily 遗留任务不受 3 条限制，项目旧欠账仍按 3 条上限控制；`今日计划` 里放今天要处理的用户新增事项和筛选后的笔记任务。
- `今日计划` 排在 `Carry-over` 前面。
- `Carry-over` 对项目旧欠账只保留 3 条最优先的任务，更多项目旧任务不在当天首页展开；daily 遗留任务不受这个上限约束。
- 如果今天以前的任一篇 `今日计划` 还有未完成项，至少保留 1 条到今天的 `Carry-over` 或 `今日计划` 中，并优先选最近一次遗留的任务。
- 如果某个阶段计划里同时存在更早的逾期任务和今天到期任务，优先把更早的逾期任务放进 `Carry-over` 或 `今日计划` 的前列。
- `今日计划` 的排序顺序应优先体现：笔记旧欠账 > 笔记今日任务 > 用户今日任务。
- 为每条任务附上计划时间：优先写 `开始时间` 和 `预计完成时间`，或写 `预计耗时`。
- 默认使用 skill 被调用的时间作为计划开始时间；如果用户在任务中明确写了时间或时长，则优先沿用用户给出的信息。
- 未明确时长的任务由模型估算，估算结果应尽量保守。
- 默认午休时段为 11:50-14:00；如果用户给出午休、会议、通勤或其他不可执行时间段，自动把后续计划顺延到下一个可执行时间点。
- 如果单个任务跨越午休等阻塞窗口，可以拆成两个连续片段排进计划，并在任务文本中保留同一任务上下文。
- 每个任务尽量保留来源信息或简短上下文，便于回溯到原笔记。
- 不要把已完成任务重新写成未完成。
- 不要改动源笔记里的任务状态，只在今日计划里做镜像和编排。

## 任务收集

- 优先使用 `scripts/collect_daily_plan_candidates.py` 拉取候选任务清单，再做筛选和重排。
- 如果脚本不可用，就按同样规则直接扫描笔记。
- `- [ ]` 视为待办，`- [x]` 视为完成。
- 保留任务里的日期标记，例如 `📅 YYYY-MM-DD`。
- `20 Projects/高软-系统架构师/任务看板.md` 是高优先级入口，但不要只依赖它。
- 默认不要扫描 `20 Projects/Stock Prompt+report` 和 `20 Projects/龟龟策略`，除非用户明确要求。

## 示例输出

```md
## 今日计划

- [ ] 10:25-11:50 + 14:00-14:35 | 预计 2h | 湖南移动网安 EU 拨测指令对接外呼平台评估，脚本改造。
- [ ] 14:35-16:35 | 预计 2h | 总结 OKR 报告。

## Carry-over

- [ ] 16:35-18:35 | 预计 2h | [[2026-04-27]]：查看代码上线事件没有处理的 bug，先对比代码，重点覆盖多采集设备的场景。
- [ ] 18:35-19:20 | 预计 45m | [[20 Projects/高软-系统架构师/00 计划/系统架构师冲刺计划-阶段一|系统架构师冲刺计划：阶段一]]：周三章节复习 - 第 3 章 计算机网络 📅 2026-04-15
```

## 冲突处理

- 如果用户输入和笔记任务冲突，先以用户明确要求为准。
- 如果信息不足，做出合理假设，并在 `Notes` 或 `Review` 中标注。
- 如果当天没有足够任务，仍然根据用户输入生成一个可执行的最小计划。
