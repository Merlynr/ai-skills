# grill-pro：L 级多角度需求拷问

**来源**：个人经验 §二「AI 多角度提问」；复杂需求提示词（架构/测试/运维/安全/产品）。

在 [grill-lite.md](../merlynr-dev-stack/grill-lite.md) 之上，为 **L 级** 提供更深的需求澄清。不替代 `gsd-discuss-phase`；12 轮仍不够时进 discuss。

> **与 grill-lite 关系**：M 级继续用 grill-lite（≤3 轮）；L 级需求未清或需业务/数据流/多角度分析时用本文。

## 何时启用

| 级别 | 用 grill-lite | 用 grill-pro |
|------|---------------|--------------|
| **S** | 否 | 否 |
| **M** | 验收略模糊 | 否（升 L 或进 discuss） |
| **L** | 可先 lite 再 pro | 多模块 / 验收模糊 / 用户要 no-code-first |

**L 级 no-code-first**：首轮 **禁止写代码、禁止给完整方案**；只提问 + 摘要，直到出口 Handoff。

## 规则（继承 grill-lite + 扩展）

1. **一次一问**；L 级最多 **12 轮**（第 8 轮起提示剩余轮次）。
2. 证据可查 → 先 M1 探库，把事实写进摘要再问缺口。
3. 每 **4 轮** 输出共识摘要（含模块候选草案）。
4. 用户说「够了」「开始吧」→ 立即停止，输出 **Grill-Pro 共识** Handoff。

## 问题维度

先覆盖 grill-lite 6 维（目标/范围/验收/约束/风险/偏好），已清楚则跳过。再按序覆盖：

| # | 维度 | 示例问题 |
|---|------|----------|
| 7 | **架构** | 与现有模块边界？依赖方向？不能耦合哪些层？ |
| 8 | **测试** | 怎样独立验证每模块？集成点测什么？失败如何复现？ |
| 9 | **运维** | CLI/热更新/回滚？线上如何观测？ |
| 10 | **安全** | 信任边界、输入校验、权限、敏感数据路径？ |
| 11 | **产品** | 用户可见行为？与现有功能冲突？ |

### 业务流三维（L 级必覆盖至少一项缺口）

| 维度 | 确认内容 |
|------|----------|
| **业务流程** | 谁触发、步骤顺序、成功/失败分支 |
| **数据流** | 数据从哪来、经谁处理、存哪、谁消费 |
| **异常与边界** | 非法输入、超时、并发、资源耗尽 |

## 出口：Grill-Pro 共识 Handoff

进入 M2（module-patterns）或 `gsd-discuss-phase` 前 **必须** 输出：

```markdown
## Grill-Pro 共识（YYYY-MM-DD）

- **级别**: L
- **no-code-first**: 是 | 已解除
- **已确认**: …
- **仍待定**: …（若无写「无」）
- **建议默认**: …

### 业务流摘要
- **业务流程**: …
- **数据流**: …
- **异常/边界**: …

### 多角度覆盖
- 架构: ✓ | 跳过（原因）
- 测试: ✓ | 跳过
- 运维: ✓ | 跳过
- 安全: ✓ | 跳过
- 产品: ✓ | 跳过

### 模块候选（草案）
| 模块 | 一句话职责 |
|------|------------|
| … | … |

- **Playbook 倾向**: 插件型 | 规则引擎型 | 流量分派型 | 自定义 | 待定
- **接口冻结**: 待 M2.5 | 不需要
- **验证策略**: 单元 | 集成 | 系统（C2 信号：…）
- **Agent 分工建议**: 设计 / 接口 / 实现 / 测试 / Review / 优化（见 agent-roles.md）
- **下一步**: module-patterns | gsd-discuss-phase | gsd-spec-phase
```

`gsd-discuss-phase` 收到后：**只追问「仍待定」**；模块候选作为 M2 输入，不重复已确认项。

可选：同步写入 [merlynr-handoff.json](handoff-schema.md)（G2）。

## 与 merlynr-dev 下游的衔接

```text
Grill-Pro Handoff
  → module-patterns.md（模块清单定稿）
  → interface-first.md（M2.5 冻结）
  → gsd-spec/plan 或 M3
```

Handoff 中 **模块候选** 应在 M2 定稿为正式模块清单；**验证策略** 若含 C2 信号，M4 必须叠加 [system-validation.md](system-validation.md)。

## 反模式

- L 级第一轮就输出大段代码或完整 design.md（no-code-first 未解除）
- 连发多题、不等待回答
- 用 grill-pro 代替 discuss（大块未知应进 `gsd-discuss-phase`）
- Handoff 无模块候选、无验证策略
