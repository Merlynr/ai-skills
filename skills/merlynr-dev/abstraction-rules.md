# 适度重复、谨慎抽象

**来源**：个人经验 §三.2；三案例共同方法 #3。

AI 更擅长复制、修改和局部优化，不擅长复杂继承、深层抽象和跨模块联动。抽象的目标是 **降低复杂度**，不是消除所有重复。

## 三条规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | **局部小逻辑允许重复** | 两个差异较大的解析器/协议/分支，先独立实现 |
| 2 | **大且通用才封装** | 能力稳定、多处复用、边界清晰 → 独立模块 + 标准 API |
| 3 | **出现明确复用需求再抽象** | 第三处相似需求或用户明确要求时再抽基类/框架 |

## 案例锚点

```text
GBT32960Parser  与  JT808Parser  → 先独立，不急于统一基类
插件 API / matcher 库 / Dispatch 规则接口 → 大且通用，值得封装
```

## 不可重复类型（禁止用「适度重复」豁免）

下列逻辑 **不得** 在多模块间复制粘贴；须抽 common / policy-common / 已有 util，或走 D2 统一改一处：

| 类型 | 示例 | 原因 |
|------|------|------|
| **内存 / 所有权** | mbuf refcnt、alloc/free、transfer、drain | 一处漏改 → 全库泄漏或 double-free |
| **锁 / 并发** | 锁顺序、per-thread 结构、atomic 边界 | AI 跨模块一致性重构极弱 |
| **解析 / 安全边界** | 长度校验、溢出防护、认证/token 校验 | CVE 需 N 份同步修复 |
| **错误码 / 生命周期契约** | public 错误码语义、对象 create/hold/release | 调用方假设不一致 |

**允许重复** 的典型：运营商字段映射、UI 配置分支、差异 >30% 的协议字段解析（见规则 #1）。

## 按级别的执行要求

| 级别 | 要求 |
|------|------|
| **S** | 跳过 |
| **M** | 大 diff 或新增类型层次前 **自检**（见下表） |
| **L** | M3 前必读；Phase 4.5 `## Decisions` + `## Similar Logic Registry`（见下） |

## M3 自检（implement 前）

| 问题 | 若答「是」则暂停抽象 |
|------|----------------------|
| 两处逻辑差异是否 >30% 行为不同？ | 先独立实现 |
| 抽象是否需要 >3 层继承或跨 >2 模块？ | 拆小任务或延后 |
| 抽象是否主要为「看起来统一」？ | 允许重复 |
| 上下文是否会因抽象增加 >5 个额外文件？ | 评估是否值得 |

## 与 stack Phase 3 的关系

stack 要求「复用项目现有命名、工具、错误处理模式」——**不是**「强行抽公共基类」。

- ✅ 复用：错误码风格、日志宏、CLI 框架、已有 util
- ❌ 过度：为两个 demo 协议抽 `AbstractParser` 基类

## 写回模板（L 级 Decisions）

```markdown
## Decisions
- **D{n} 不抽象 {A}/{B}**：两者 {差异点}；第三处复用未出现；证据：{文件:符号}
- **D{n} 封装 {模块}**：{N} 处调用、接口已冻结；证据：INTERFACE.md §…
```

## Similar Logic Registry（L 级 Phase 4.5 必须）

记录 **刻意保留的相似逻辑**，便于未来安全/性能类 bug 一次性排查，而非全库盲搜。

```markdown
## Similar Logic Registry

| 逻辑域 | 模块 A | 模块 B | 为何未抽象 | 统一修复触发 |
|--------|--------|--------|------------|--------------|
| policy 字段解析 | policy-cmcc | policy-ctcc | 字段差异 >30% | 第三处运营商或安全类 bug |
| … | … | … | … | 第三处相似需求 / CVE / 用户要求收敛 |

- **登记规则**：凡 Decisions 中「不抽象」项 **必须** 在此表有一行；无重复则写「无（本任务未保留跨模块重复）」。
- **M 级**：跨模块且存在「不抽象」取舍时 **建议** 写入主模块 `AGENTS.md` 同表。
- **收敛触发**：第三处相似需求、安全类漏洞、或用户明确要求 → 开抽象任务（非 silent copy-fix）。
```

## 反模式

- 为两个差异协议过早抽统一基类
- 为统一而拉长上下文、增加跨模块耦合
- 在 Handoff 未稳定时做全局 refactor
- 把「禁止重复」当作工程美德
- 内存/锁/解析边界在多模块复制并标为「适度重复」
- L 级有「不抽象」决策却不登记 Similar Logic Registry
