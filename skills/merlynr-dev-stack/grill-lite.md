# grill-lite：轻量需求拷问

借鉴 L 站 grill-me，但 **不创建 Trellis task、不替代 GSD discuss-phase**。
用于：想法模糊、验收标准缺失、用户说「你先看着办」。

## 规则

1. **一次一问**，等用户答完再问下一题。
2. **最多 12 轮**；第 8 轮起每轮提示「还可追问 N 轮」。
3. **禁止**在证据可查时空问——先探库/检索，把已知事实写进摘要再问缺口。
4. 每 4 轮输出 **共识摘要**（已确认 / 仍待定 / 建议默认）。
5. 用户说「够了」「开始吧」→ 立即停止拷问，进入 GSD 或执行。

## 问题优先级

按顺序覆盖，已清楚的跳过：

| # | 维度 | 示例问题 |
|---|------|----------|
| 1 | 目标 | 做完后用户/系统能做什么新事？ |
| 2 | 范围 | 明确不做什么？ |
| 3 | 验收 | 怎样算完成？可执行的检查命令？ |
| 4 | 约束 | 性能/兼容/不能动的模块？ |
| 5 | 风险 | 破坏性操作、数据迁移、线上影响？ |
| 6 | 偏好 | 实现风格：最小 diff vs 顺便重构？ |

## 出口决策

| 状态 | 下一步 |
|------|--------|
| 目标+验收清晰 | S/M → 直接做；L → `gsd-plan-phase` |
| 仍有大块未知 | `gsd-discuss-phase` 或 `gsd-explore` |
| 仅缺外部事实 | 先 librarian / browser / WebFetch，再决定 |

## 反模式

- 不要连发 5 个问题
- 不要把拷问当拖延——S 级任务跳过本流程
- 不要替用户做不可逆决定（删库、force push、改生产配置）
