# AI 编程案例分享

这三个案例不是简单让 AI 写几个函数，而是用 AI 辅助完成了三个具备工程边界的模块：

1. 插件注册和管理模块
2. 精细化规则匹配引擎及接口
3. Dispatch 流量分派模块

它们可以说明一件事：只要模块边界拆得清楚、接口定义明确、验证手段跟上，AI 可以参与完成较复杂的工程级开发任务。

## 二、案例一：插件注册和管理模块

### 1. 背景

这个模块的目标是把插件开发从“依赖主工程经验”变成“有接口、有示例、有文档、有运行时管理”的标准化能力。

插件开发者只需要理解开放的插件 API，就可以开发 `.so` 插件，并通过配置或命令加载到主程序中。

### 2. 规模

| 统计维度 | 文件数 | 行数 |
| --- | ---: | ---: |
| 总计 | 24 | 3858 |
| C 源码 | 13 | 2628 |
| 头文件 | 3 | 228 |
| Markdown 文档 | 5 | 766 |
| Shell / YAML / Makefile | 3 | 236 |

其中 demo 目录覆盖了 HTTP、DNS、SNMP、SSH、stream 插件等多种示例。

### 3. 设计亮点

- **统一插件 API**：插件通过统一入口注册，不需要了解主程序全部内部细节。
- **动态加载和运行时管理**：支持插件加载、更新、启用、禁用、查看状态。
- **HTTP 事件回调模型**：插件只关注请求头、请求体、响应头、响应体、事务结束等事件点。
- **事务级状态管理**：每个 HTTP transaction 可以维护独立状态，适合请求响应关联检测。
- **标准化开发套件**：通过脚本打包编译环境、头文件、示例代码和说明文档。

### 4. 实际效果

这个案例证明，AI 不只适合生成检测代码，也可以辅助建设一套可复用的插件开发基础设施。

更重要的是，它把后续插件开发的门槛降下来了。开发者不需要读完整主工程，只要读插件文档和 demo，就可以完成开发、编译、部署和调试。

PDF 中的插件开发说明也体现了完整流程：

```text
打开开发环境
  ↓
阅读接口文档和 demo
  ↓
让 AI 先设计再编码
  ↓
编译生成插件
  ↓
放入插件目录
  ↓
通过 CLI 加载、更新、查看状态
  ↓
根据运行结果继续让 AI 辅助调试
```

这个闭环能增强同事使用 AI 编程的信心：AI 不是只能写一次性代码，而是可以在明确规则下持续迭代工程模块。

## 三、案例二：精细化规则匹配引擎及接口

### 1. 背景

`jxh-rule` 模块是一套接入 Perseus 流量分析系统的精细化规则匹配与输出模块。

它不是单点功能，而是覆盖规则加载、规则匹配、热更新、Flow 生命周期、多输出链路和 CLI 运维的一整套能力。

### 2. 规模

| 统计口径 | 文件数 | 行数 |
| --- | ---: | ---: |
| 目录全量 | 120 | 59468 |
| 源码相关口径 | 117 | 27247 |
| C/H 代码 | 76 | 24022 |

关键文件包括：

| 文件 | 行数 | 说明 |
| --- | ---: | --- |
| `jxh-rule-mgr.c` | 2737 | 规则管理器核心 |
| `src/frame/index.c` | 1774 | matcher 索引实现 |
| `src/frame/matcher.c` | 1601 | 规则匹配主引擎 |
| `src/frame/xml_parser.c` | 1473 | 规则 XML 解析 |
| `test_comprehensive_integration.c` | 1100 | 综合集成测试 |

### 3. 设计亮点

- **规则运行时和业务输出解耦**：输出模块调用统一接口，不关心规则加载和更新细节。
- **热更新具备回滚能力**：新规则构建、第三方规则导出、文件替换、编译或 reload 任一步失败，都能回滚到旧状态。
- **跨线程生命周期内存管理**：Flow 上下文通过 transfer 对象安全转交给输出线程，再由消息释放机制统一回收。
- **多链路统一接入**：HTTP、普通流、文件还原、事件日志都能进入统一规则匹配体系。
- **可观测和可运维**：支持状态查看、命中统计、规则查询、性能统计、手动更新和内存域排障。

### 4. 实际效果

这个案例最能体现 AI 编程在大规模工程中的价值。

AI 主要帮助完成：

- 大量样板代码生成
- 多文件接口联动
- 错误码、状态机、CLI 框架补齐
- 字段映射和输出结构整理
- 文档、测试和辅助工具铺开

人工重点把控：

- 模块边界
- 生命周期安全
- 与原系统线程模型的兼容性
- 热更新失败后的回滚逻辑
- 关键路径性能和可靠性

这个案例说明：AI 可以参与上万行工程模块开发，但前提是人要先把模块边界、接口和验证方式定义清楚。

## 四、案例三：Dispatch 流量分派模块

### 1. 背景

Dispatch 模块用于在主程序处理流量的同时，根据 MSISDN、IPv4、IPv6 等规则，把指定流量分发到不同的 DPDK ring，供车联网、物联网、加密流量分析、数据包留存等专项程序消费。

它的核心价值是把主程序和专项分析程序解耦。

### 2. 规模

| 文件 | 行数 | 主要职责 |
| --- | ---: | --- |
| `dispatch-rule.c` | 1426 | 规则加载、三表规则库、区间合并、二分匹配、热更新、flow 缓存 |
| `tmodule-dispatch-ring.c` | 1525 | 分派主逻辑、ring 创建、action 执行、CLI、统计、mbuf 生命周期 |
| `message-output-match-info.c` | 600 | 命中信息聚合、用户信息补充、流量统计输出 |
| `tmodule-dispatch-recv.c` | 425 | secondary 接收侧批量收包 |
| 文档和展示材料 | 2290 | README、设计说明、技术介绍、产品介绍 |
| 合计 | 5880 | 完整 Dispatch 子系统 |

### 3. 设计亮点

- **规则模型清晰**：MSISDN、IPv4、IPv6 拆成三张表，避免不同类型规则混用。
- **热路径性能优化**：规则排序合并后用二分匹配，flow 级缓存减少重复查询。
- **规则热更新低阻塞**：锁外构建新规则库，短写锁替换，锁外释放旧规则库。
- **分派动作灵活**：支持 drop、dispatch、mirror-ref、mirror-copy。
- **mbuf 生命周期明确**：不同 action 下明确所有权、refcnt、copy、free 和 drain 规则。
- **运维闭环完整**：支持状态查看、统计清理、规则更新和命中查询。

### 4. 实际效果

这个案例说明 AI 不只适合业务代码，也可以辅助高性能系统模块开发。

AI 的价值主要体现在：

- 快速搭建规则引擎、ring 分派、接收模块和 CLI 骨架
- 帮助补齐 ring full、非法规则、reload、refcnt、退出 drain 等边界条件
- 持续辅助性能优化，例如 flow 缓存、本地统计回刷、mirror 高水位采样
- 同步生成 README、设计说明和产品介绍材料

这个模块最终形成了从规则配置、流量分派、专项消费、统计输出到运维命令的完整闭环。

## 五、三个案例共同体现的 AI 编程方法

### 1. 从设计阶段就开始模块化

这三个案例都不是先让 AI 写代码，再后期整理结构，而是从需求阶段就先拆模块：

- 插件模块拆成 API、管理器、HTTP 回调、事件输出、demo、文档。
- 规则模块拆成规则管理器、matcher、Flow 上下文、输出链路、内存管理、CLI。
- Dispatch 模块拆成规则引擎、分派执行、接收侧、统计输出、运维命令。

模块越小、边界越清楚，AI 越容易在小上下文里稳定产出。

### 2. 先定义接口，再分模块实现

AI 编程最怕的是目标模糊和上下文过大。

比较有效的方式是：

```text
先定义模块职责
  ↓
再定义输入输出
  ↓
再定义错误码、状态机和生命周期
  ↓
最后让 AI 分模块实现
```

这样生成的代码更容易集成，也更容易测试和 Review。

### 3. 适度重复优于过早抽象

在这些案例里，并不是所有相似逻辑都强行抽象成统一基类或统一框架。

对于局部、小型、差异较大的功能，允许适度重复，反而能让每个模块更简单、更清楚，也更适合 AI 理解和修改。

只有当能力较大、稳定且通用时，才封装成独立模块和标准 API，例如插件 API、matcher 库、Dispatch 规则接口。

### 4. AI 负责提效，人负责兜底

AI 在这些案例里主要提高了实现效率：

- 生成大段样板代码
- 补齐接口和结构体
- 整理重复字段
- 生成文档和测试
- 快速迭代边界条件

但关键判断仍然由人把控：

- 模块边界是否合理
- 生命周期是否安全
- 热更新失败能否回滚
- 性能路径是否可接受
- 真实环境是否验证通过

## 六、得出的几个结论

1. AI 编程不是只能写 demo，也能参与真实工程模块。
2. AI 更适合边界清晰的小上下文任务，所以需求阶段就要开始模块化。
3. 一定范围内允许重复造小轮子，比过早全局抽象更适合 AI 协作。
4. 复杂任务可以拆成多个 Agent 角色：设计、实现、测试、Review、优化。
5. AI 能提高编码和文档产出速度，但上线质量仍依赖工程师的设计、判断和验证。

一句话总结：

> AI 编程的关键不是让 AI 一口气写完整个系统，而是把复杂工程拆成 AI 能稳定完成的小任务，再由工程师把控边界、集成和验证。
